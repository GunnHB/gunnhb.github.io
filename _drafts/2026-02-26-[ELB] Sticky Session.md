## Sticky Session
**고정 세션**<sup>Sticky Session</sup>은 로드 밸런서를 통해 들어오는 사용자의 트래픽을 처리할 때 ***특정 사용자의 요청을 항상 처음 연결되었던 동일한 서버로만 보내는 방식***이다.

보통 트래픽을 분산시키기 위해 여러 대의 서버를 운영한다. 사용자가 로그인을 하거나 장바구니에 물건을 담으면 이 세션 정보는 기본적으로 요청을 처리한 특정 서버의 로컬 메모리에 저장된다.

만약 첫 번째 요청(로그인)을 `서버 A`가 처리했는데 두 번째 요청(상품 결제)을 로드 밸런서가 `서버 B`로 보낸다면 로그인 사실을 모르기 때문에 갑자기 로그아웃이 되거나 오류가 발생한다.

이를 방지하기 위해 ***한 사용자의 요청은 무조건 하나의 서버에 고정***되도록 하는 것이 고정 세션이다.

### 동작 원리
고정 세션은 주로 **쿠키**<sup>Cookie</sup>를 활용해 구현한다.

1. 사용자가 처음 접속하면 로드 밸런서가 가장 여유로운 서버(`서버 A`)로 요청을 전달한다.
2. 응답이 나갈 때 로드 밸런서는 "이 사용자는 `서버 A` 담당"이라는 정보가 담긴 쿠키를 사용자(브라우저)에 전달한다.
3. 이후 사용자가 다시 요청을 보낸 때 이 쿠키를 함께 보내면 로드 밸런서가 쿠키를 확인하고 무조건 `서버 A`로 요청을 보낸다.

### 장점
- 단순한 구조
  - 세션 정보를 여러 서버가 공유하도록 복잡하게 시스템을 뜯어고칠 필요가 없다.

- 빠른 속도
  - 각 서버가 자신의 메모리에 있는 세션 데이터를 바로 읽어오기 때문에 처리가 빠르다.

### 단점
- 트래픽 불균형 발생
  - 특정 서버에 할당된 사용자들이 유독 활발하게 활동하는 경우 해당 서버만 과부하가 걸리게 된다.

- 가용성 저하
  - 서버에 장애가 발생할 경우 고정 세션으로 연결되었던 사용자의 정보가 모두 날아간다.


## 쿠키
**쿠키**<sup>Cookie</sup>를 통해 사용자는 특정 서버에 고정된다.
