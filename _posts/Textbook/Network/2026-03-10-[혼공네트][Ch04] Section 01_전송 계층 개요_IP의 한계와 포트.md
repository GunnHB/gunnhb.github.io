## 신뢰할 수 없는 통신과 비연결형 통신
IP는 **신뢰할 수 없는 프로토콜**<sup>unreliable protocol</sup>이자 **비연결형 프로토콜**<sup>connectionless protocol</sup>이라는 한계가 있다.

- **신뢰할 수 없는 통신** 
  - IP 프로토콜이 패킷이 수신지까지 제대로 전송되었다는 보장을 하지 않는다.
    - 통신 과정에서 패킷의 데이터가 손상되거나 중복된 패킷이 전송되었더라도 이를 확인하지 않고,<br>
      재전송도 하지 않으며, 순서대로 패킷이 도착할 것이라는 보장도 하지 않는다는 의미
  - **최선형 전달**<sup>best effort delivery</sup>이라고도 부른다.

- **비연결형 통신**
  - 송수신 호스트 간에 사전 연결 수립 작업을 거치지 않는다.
    - 단순히 수신지를 향해 패킷을 보내기만 한다.
  - 패킷의 **빠른 송수신**을 위함

## IP의 한계를 보완하는 전송 계층
1. 연결형 통신을 지원
    - 송수신하는 동안 연결을 유지하고 끝나면 연결을 종료

2. 신뢰성 있는 통신이 가능
  - 패킷이 순서대로 전달되는 것을 보장하기 위해 다양한 기능들을 제공

## 응용 계층과의 연결 다리, 포트
### 포트의 정의
- **포트**<sup>port</sup>
  - 패킷이 실행 중인 특정 애플리케이션까지 도달하기 위한 식별 정보

### 포트의 분류
전송 계층에서는 **포트 번호**를 통해 특정 애플리케이션을 식별한다.

|포트 종류|포트 번호 범위|
|---|---|
|잘 알려진 포트<sup>well known port</sup>|0-1023|
|등록된 포트<sup>registered port</sup>|1024-49151|
|동적 포트<sup>dynamic port</sup>|49152-65535|

**잘 알려진 포트**<sup>well known port</sup>는 범용적으로 사용되는 애플리케이션 프로토콜의 포트 번호를 의미한다.

|잘 알려진 포트<sup>well known port</sup>|설명|
|---|---|
|20, 21|FTP|
|22|SSH|
|23|TELNET|
|53|DNS|
|67, 68|DHCP|
|80|HTTP|
|443|HTTPS|

**등록된 포트**<sup>registered port</sup>는 잘 알려진 포트에 비해 덜 범용적이지만 흔히 사용되는 애플리케이션 프로토콜의 포트 번호를 의미한다.

|등록된 포트<sup>registered port</sup>|설명|
|---|---|
|1194|OpenVPN|
|1433|Microsoft SQL Server 데이터베이스|
|3306|MySQL 데이터베이스|
|6379|Redis|
|8080|HTTP 대체|

**동적 포트**<sup>dynamic port</sup>는 **사설 포트**<sup>private port</sup>, **임시 포트**<sup>ephemeral port</sup>라고도 불리며 할당된 애플리케이션 프로토콜이 없고 특별히 관리되지 않는 포트 번호인 만큼 자유롭게 사용 가능하다.

서버로서 동작하는 프로그램은 일반적으로 잘 알려진 포트와 등록된 포트로 동작하는 경우가 많다. 반면에 클라이언트로서 동작하는 프로그램은 동적 포트 번호 중 임의의 번호가 할당되는 경우가 많다.

**특정 호스트에서 실행 중인 특정 애플리케이션 프로세스**는 IP 주소와 포트 번호를 통해 `IP 주소`:`포트 번호` 형태로 식별할 수 있다.

## 포트 기반 NAT
- **NAT**
  - 사설 IP 주소와 공인 IP 주소를 변환
  - IP 변환은 **NAT 변환 테이블**을 통해 수행
    - 네트워크 내부의 사설 IP와 외부의 공인 IP 주소의 쌍
    - 사설 IP와 공인 IP가 일대일로 매칭된다면 사설 IP 수만큼 공인 IP가 필요
    - 오늘날 NAT 기술은 **포트를 통해** 대부분 다수의 사설 IP 주소를 그보다 적은 수의 공인 IP 주소로 변환

|네트워크 외부|네트워크 외부|
|---|---|
|1.2.3.4|192.168.0.5|
|1.2.3.5|192.168.0.6|
|...|...|

### NAPT
**NAPT**<sup>Network Address Port Translation</sup>
  - 포트 기반의 NAT
  - 하나의 공인 IP 주소를 여러 사설 IP 주서가 공유할 수 있도록 하는 NAT의 일종

|네트워크 외부|네트워크 외부|
|---|---|
|1.2.3.4:6200|192.168.0.5:1026|
|1.2.3.4:6201|192.168.0.6:1026|
|...|...|

네트워크 외부에서 사용할 IP 주소가 같더라도 포트 번호가 다르면 네트워크 내부의 호스트를 특정할 수 있어 다수의 사설 IP 주소를 그보다 적은 수의 공인 IP 주소로 변환할 수 있다.

## 좀 더 알아보기
### 포트 포워딩
- **포트 포워딩**<sup>port forwarding</sup>
  - 네트워크 내 특정 호스트에 IP 주소와 포트 번호를 미리 할당하고 해당 `IP 주소`:`포트 번호`로써 해당 호스트에게 패킷을 전달하는 기능
  - 네트워크 내부의 여러 호스트가 같은 공인 IP를 공유하는 경우
    - 처음 외부에서 내부로 통신을 시도할 때 내부의 어느 사설 IP를 수신지로 삼을지 결정이 어려움
    - 어떤 `공인 IP 주소`:`외부 접속 포트`에 접근했을 때 어떤 `사설 IP 주소`:`내부 전달 포트`로 전달할지 설정 가능

### ICMP
- **ICMP**<sup>Internet Control Message Protocol</sup>
  - IP의 신뢰할 수 없는 전송 특성과 비연결형 전송 특성을 보완하기 위한 **네트워크 계층**의 프로토콜
  - IP 패킷의 전송 과정에 대한 피드백 메시지(ICMP 메시지)를 얻기 위해 사용하는 프로토콜
  - 메시지 종류
    - 전송 과정에서 발생한 문제 상황에 대한 오류 보고
    - 네트워크에 대한 진단 정보
  - ICMP 메시지는 타입과 코드로 정의
    - ICMP 패킷 헤더에 포함되어 있는 정보
    - 타입<sup>type</sup> 필드
      - ICMP 메시지의 유형 (번호로 명시)
    - 코드<sup>code</sup> 필드
      - 구체적인 메시지 내용 (번호로 명시)

- **유의할 점**
  - IMCP는 어디까지나 IP의 신뢰할 수 없는 ***특성을 보완하기 위함***인 것이지 **신뢰성을 보장하는 것은 아님**
  - 신뢰성을 완전히 보장하기 위해서는 전송 계층의 프로토콜이 필요
