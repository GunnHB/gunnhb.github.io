## 데이터 링크 계층의 한계
물리 계층과 데이터 링크 계층만으로 LAN을 넘어서 통신하기 어려운 두 가지 이유가 있다.

### 다른 네트워크까지의 도달 경로를 파악하기 어렵다
서로 다른 LAN에 속한 호스트들이 메시지를 주고받는다면 수많은 네트워크 장비를 거쳐야한다. 통신을 빠르게 주고받으려면 최적의 패킷 경로를 찾아야 하는데 이것을 **라우팅**<sup>routing</sup>이라고 한다. 이는 **라우터**<sup>router</sup>라는 네트워크 계층의 장비로 수행된다.

### MAC 주소만으로는 모든 네트워크에 속한 호스트의 위치를 특정하기 어렵다
MAC 주소는 일종의 `수신인`이다. 메시지를 보낼 때 수신인의 정보만을 가지고 처리할 순 없다. 어디에 위치하고 있는지는 모르기 때문이다. 이때 `수신지` 역할을 하는 정보가 네트워크 계층의 **IP 주소**이다.

> IP 주소는 **논리 주소**라고도 불린다. 이는 **DHCP**<sup>Dynamic Host Configuration Protocol</sup>라는 특정 프로토콜을 통해 자동으로 할당받거나 직접 할당할 수 있다.
{: .prompt-tip}

## 인터넷 프로토콜
**인터넷 프로토콜**<sup>IP: Internet Protocol</sup>은 네트워크 계층의 핵심적인 프로토콜 중 하나이다.

### IP 주소 형태
IP 주소는 4바이트(32비트)로 표현된다. 숫자당 8비트로 표현되기에 0-255 범위 안에 있는 네 개의 10진수(**옥텟**<sup>octet</sup>)로 표기된다.

> 192.168.1.1

### IP의 기능
대표적인 IP의 기능에는 **IP 주소 지정**과 **IP 단편화**가 있다.

**IP 주소 지정**<sup>IP addressing</sup>은 IP 주소를 바탕으로 송수신 대상을 지정하는 것을 의미한다. **IP 단편화**<sup>IP fragmentation</sup>는 전송하고자 하는 패킷의 크기가 **MTU**<sup>Maximum Transmission Unit</sup>라는 최대 전송 단위보다 클 경우 이를 MTU 크기 이하의 복수의 패킷으로 나누는 것을 의미한다.

> MTU란 한 번에 전송 가능한 IP 패킷의 최대 크기이다. 일반적으로 1500바이트이며 크기 이하로 나누어진 패킷은 수신지에서 다시 재조합된다.
{: .prompt-tip}

### IPv4
IPv4 패킷 헤더의 주요 필드에는 `식별자`, `플래그`, `단편화 오프셋`, `TTL`, `프로토콜`, `송신지 IP 주소`, `수신지 IP 주소`가 있다.

- 식별자<sup>identifier</sup>
  - 패킷에 할당된 번호
  - 어떤 메시지에서부터 쪼개져 보내진 메시지인지 식별하기 위한 필드

- 플래그<sup>flag</sup>
  - 세 개의 비트로 구성된 필드
  - 첫 번째 비트는 0으로 고정 (미사용되는 비트)
  - 두 번째 비트(**DF**<sup>Don't Fragment</sup>)는 IP 단편화를 수행하지 말라는 표시
  - 세 번째 비트(**MF**<sup>More Fragment</sup>)는 단편화된 패킷이 더 있는지 표시

- 단편화 오프셋<sup>fragment offset</sup>
  - 패킷이 단편화되기 전에 패킷의 초기 데이터에서 몇 번째로 떨어진 데이터인지 표시
  - 재조합을 위한 필드

- TTL<sup>Time To Live</sup>
  - 패킷의 수명
  - 패킷이 하나의 라우터를 거칠 때마다 1씩 감소
  - 무의미한 패킷이 네트워크 상에 지속적으로 남아있는 것을 방지하기 위함

> 패킷이 호스트 또는 라우터에 한 번 전달되는 것을 **홉**<sup>hop</sup>이라고 한다.
{: .prompt-tip}

- 프로토콜
  - 상위 계층의 프로토콜이 무엇인지 나타내는 필드

- 송신지 IP 주소<sup>Source IP Address</sup> / 수신지 IP 주소<sup>Destination IP Address</sup>
  - 송수신지의 IPv4 주소를 표시

### IPv6
**IPv6**는 IPv4의 공급 수 부족(2<sup>32</sup>, 약 43억)으로 인해 등장했다. 16바이트(128비트)로 표현되며 콜론으로 구분한다.

IPv6 패킷의 헤더는 `다음 헤더`, `홉 제한`, `송신지 IP 주소`, `수신지 IP 주소`로 구성된다.

- 다음 헤더<sup>next header</sup>
  - 상위 프로토콜 또는 확장 헤더를 표시
  - 확장 헤더<sup>extension header</sup>: 추가적인 헤더 정보
    - 홉 간 옵션<sup>hop-by-hop options</sup>
    - 수신지 옵션<sup>Destination Options</sup>
    - 라우팅<sup>Routing</sup>
    - 단편<sup>Fragment</sup>
    - ESP<sup>Encapsulating Security Payload</sup>
    - AH<sup>Authentication Header</sup>

- 홉 제한<sup>hop limits</sup>
  - 패킷의 수명

- 송신지 IP 주소<sup>Source IP Address</sup> / 수신지 IP 주소<sup>Destination IP Address</sup>
  - 송수신지의 IPv6 주소를 표시

## ARP
**ARP**<sup>Address Resolution Protocol</sup>는 IP 주소를 통해 MAC 주소를 알아내는 프로토콜이다. 동일 네트워크 내에 있는 송수신 대상의 IP 주소를 통해 MAC 주소를 알아낼 수 있다.

ARP의 동작 과정은 다음과 같다.

1. ARP 요청
  - 브로드캐스트 메시지를 통해 모든 호스트에게 **ARP 요청**<sup>ARP Request</sup>이라는 ARP 패킷을 전송

2. ARP 응답 
  - 수신한 호스트 중 대상 호스트는 **ARP 응답**<sup>ARP Reply</sup>이라는 ARP 패킷을 요청 호스트에 전송

3. ARP 테이블 갱신
  - 위 과정을 통해 대상 호스트의 MAC 주소를 테이블에 기록
  - ARP를 활용할 수 있는 호스트라면 **ARP 테이블**<sup>ARP Table</sup>이라는 정보를 유지

> 다른 네트워크 내의 호스트의 MAC 주소를 알기 위해서는 라우터를 통해 같은 과정을 수행한다.
{: .prompt-tip}

## 좀 더 알아보기
### IP 단편화를 피하는 방법
IP 단편화는 되도록 하지 않는 것이 좋다. 이를 위해서는 ***IP 단편화 없이 주고받을 수 있는 최대 크기***만큼만 전송해야 한다. 이 크기를 **경로 MTU**<sup>Path MTU</sup>라고 한다. 그리고 이렇게 IP 단편화를 회피하는 기술을 **경로 MTU 발견**<sup>Path MTU discovery</sup>이라고 한다.
